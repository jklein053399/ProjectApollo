# Work Report — 2026-04-12 (Session 7)

Session 7 was the first real build session after six sessions of design. The goal was narrow but concrete: get FastAPI running on the Pi 5, deployed under systemd, surviving a reboot with a working `/health` endpoint. All done-when criteria met.

The session opened with Jacob having imaged a fresh microSD, plugged the Pi into the network, and asking me to find it. An ARP sweep of the `10.0.0.0/24` subnet surfaced `10.0.0.138` with MAC prefix `2c:cf:67:...` — a Raspberry Pi Trading OUI — and SSH port 22 open. First SSH attempt hit two rabbit holes. The host key had changed from a prior reimage (expected, resolved with `ssh-keygen -R`). Then every password attempt came back as `Permission denied`. `ssh -v` showed the server offering password auth and silently rejecting it. The eventual diagnosis: Pi Imager silently lowercases usernames in its config UI. Jacob had typed `cypherKlein`; the actual Linux account was `cypherklein`. Lowercase worked first try. That quirk is now saved in project memory.

EEPROM and boot config turned out simpler than expected. The Session 6 memory warned that the Pi 5 needed an SD-first boot to update EEPROM before USB/NVMe would work. Stale — Jacob's Pi shipped with November 2025 firmware, already supporting NVMe boot natively. `rpi-eeprom-update` confirmed current, `rpi-eeprom-config` showed `BOOT_ORDER=0xf461` (SD first). Good default for that moment since NVMe wasn't yet installed.

After Jacob installed the NVMe and `lsblk` confirmed `nvme0n1` at 238.5 GB, clean and unpartitioned, we moved to the clone. Installed `rpi-clone` from geerlingguy's GitHub fork, ran `sudo rpi-clone nvme0n1`, which partitioned, formatted, rsynced, and updated the UUIDs on `fstab` and `cmdline.txt` for the cloned partitions. The slow unmount at the end was a worthwhile teaching moment — the kernel's page cache hides gigabytes of pending writes that the unmount has to flush before detaching the filesystem. After the clone, we updated `BOOT_ORDER` to `0xf416` (NVMe → SD → USB → loop). A confusing moment: the post-write read-back still showed the old value, because the Pi 5 bootloader caches config via a `blconfig` device that only refreshes at boot. After reboot, `findmnt /` showed `/dev/nvme0n1p2` as root — NVMe was now primary, SD kept in place as passive fallback.

Midway through the session Jacob asked for every command to be explained in depth — what it does, each flag, why, expected output. Saved to memory as `feedback_teaching_mode.md`. That framing shaped the rest of the session.

FastAPI deployment was the main event. The Windows repo already had a scaffold from April 5: `src/main.py` with a `/health` endpoint, `src/config.py` using pydantic-settings, `requirements.txt` pinning `fastapi>=0.115`, `uvicorn[standard]>=0.32`, `pydantic-settings>=2.6`, plus `.env.example`. We `scp -r`'d those to `/home/cypherklein/apollo/`, cleaned up the `__pycache__` that hitchhiked along (Windows Python 3.14 bytecode, wrong for the Pi's 3.13.5), normalized the bizarre `drwx---rwx` permissions that scp applied, created a venv, `pip install -r requirements.txt`, and smoke-tested with `uvicorn src.main:app --host 0.0.0.0 --port 8000`. First curl from Windows returned 200 with the expected JSON including `env=production` from the `.env` file.

Then systemd. I initially dumped the full six-step procedure (unit file creation through reboot verification) in a single response. Jacob pushed back: "that's far too much to give me at once. Lets focus on 2 steps at a time moving forward." Saved to memory as `feedback_pacing.md`. Cleaned up, `systemd-analyze verify` returned clean, `systemctl daemon-reload && enable --now apollo` brought the service up. The real test came after `sudo reboot`: with no SSH login and no manual action, `curl http://10.0.0.138:8000/health` from Windows returned 200. Session 7 done-when criteria officially met.
