# Work Report — 2026-06-09 (Session 9)

Session 9 came after a two-month gap — the longest of the project — and opened with "where did we leave off?" The refresher reconstructed a fork the memory files had missed: the software track had been idle since Session 7 (April 12), while Sessions 8+ went into enclosure CAD that never got committed, ending with an undocumented `LaptopCyberdeck.blend` from mid-May. The session contract settled on the software track: get the project back to a working, iterable state. Jacob ruled the CAD work out of version control — `scad/` is now gitignored, and the rest of the untracked work is a parked decision.

The Pi turned out to be the easy part: first probe after two months idle returned `/health` 200 — `apollo.service` doing exactly what Session 7 built it to do. Everything else was about the loop around it.

SSH key auth ate the middle of the session, in three acts. Act one: the textbook `Get-Content | ssh "cat >> authorized_keys"` install silently corrupted the key file — Windows PowerShell appends CRLF to piped output, and the stray `\r` breaks the key line. Act two: the password fallback turned out to be a wall; no record of the Pi password existed anywhere (repo, memory, Pi Imager's saved settings — checked all three). Jacob brute-forced his own memory: the auth log later showed four failures, then `Accepted password` at 19:31:58 on candidate five. The key went in cleanly from inside that session. Act three: after getting "booted out," every password failed again and a manual reset looked inevitable — until the log showed `invalid user cypherKlein`. The capital K. The same Pi-Imager-lowercases-usernames quirk that cost Session 7 an hour, back for a second round despite being documented in three places. The structural fix: a Windows `~/.ssh/config` alias (`ssh cyberdeck`) that bakes in the lowercase username, IP, and key — documentation failed twice, so the username is no longer typed at all. One wrinkle: Windows OpenSSH rejected the new config file over an inherited orphaned-SID ACL entry; `icacls /inheritance:r /grant:r` fixed it. Password epilogue: reset at session close to one of Jacob's main known passwords.

The deploy workflow swap was quick by comparison. A drift check (throwaway clone, `diff -r` against the live tree) confirmed the Pi's code matched origin/main except `__pycache__`, so `~/apollo` was converted in place to a git checkout — init, remote add, fetch, hard reset, upstream — preserving `venv/` and `.env`, and finally evicting a stray file literally named `EOF` left by Session 7's heredoc mishap. The new `deploy/deploy.sh` does pull `--ff-only`, reinstalls deps only when requirements.txt changed, restarts the service, and health-checks with connection-refused retries. Its first run produced the session's best moment: the script couldn't run because it hadn't been deployed yet — deploy tooling has a chicken-and-egg bootstrap, resolved with one manual pull. The exit criterion landed cleanly: version bumped 0.1.0 → 0.1.1 on Windows, committed, pushed, pulled, restarted, and visible in `/health` from across the room.

Notable side discoveries: the Pi user has NOPASSWD sudo (so Claude can deploy end-to-end non-interactively); the GitHub repo is public, so the Pi pulls anonymously; Bookworm wants `journalctl -u ssh`, not `-t sshd`; and Windows OpenSSH mirrors Unix strict-modes using ACLs.

By the numbers: 3 commits to main (gitignore housekeeping, v0.1.1 + deploy script, session docs), 4 of 4 session tasks completed, 1 keypair generated, 2 lockouts diagnosed via auth log, 5 password candidates to one hit, 0 SD-card surgeries required, and a 2-month-idle service that never missed a request. The dev loop that Session 7 left as "needs upgrade" is now: edit → commit → push → `ssh cyberdeck "bash ~/apollo/deploy/deploy.sh"`. Session 10 starts at the T0 keyword dispatcher with zero infrastructure debt in the way.
